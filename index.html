A developer who thinks in systems

I started building things on the web before I knew what a framework was — raw HTML, trial and error, real problems. That foundation still shapes how I work: start with the problem, then pick the tool.

Over three years I've moved across the stack — from pixel-perfect frontends to database schemas, from configuring Linux servers to flashing microcontroller firmware. The discipline I keep returning to is systems thinking: understanding how parts connect before you start building them.

Today I focus on fullstack web development, IoT integration, and the network infrastructure that ties it all together. I care about code that's readable, deployable, and genuinely useful to the people who use it.

02

How I think
about the work

Two principles that show up in every project.

A.01 Perspective

Understand the whole
before touching the parts

Every feature exists inside a system. Before I write a line of code I ask: what breaks if this changes? Who does this actually serve? The architecture decision made at line one is the one you'll undo at line ten thousand.

Systems Thinking Architecture First
A.02 Constellation

Small things compound
into everything

Clean function names, consistent error handling, thoughtful variable scope — none of these feel important in the moment. Together they're the difference between a codebase someone can extend and one that only the original author can navigate.

Craft Long-term Thinking
